# JS MVP интерактивного курса-конструктора

Ревизия: 2026-06-01.

Стартуем с JavaScript, потому что его можно быстро показать в браузере: блоки кода, живой runner, DOM-визуализация и мобильная версия без тяжелой инфраструктуры.

---

## 1. Продуктовая идея

Приложение учит программированию через сборку проектов из готовых блоков. Студент перетаскивает блоки как конструктор, видит сгенерированный код, запускает его и получает объяснение ошибок. Внутренняя сборка автоматизирована, но в продвинутом режиме ее можно открыть и изменить.

Главный принцип: не заменять язык визуальной игрушкой, а дать безопасный вход в реальный JavaScript, где визуальный слой постепенно снимается.

---

## 2. Режимы

### Обучающий
- Доступны только нужные блоки текущего урока.
- Код генерируется автоматически.
- Подсказки объясняют, какой блок нужен дальше.
- Ошибки переводятся на человеческий язык.
- Внутренние файлы скрыты: студент видит проект, но не тонет в конфигурации.

### Продвинутый
- Можно редактировать код напрямую.
- Открываются `package.json`, `.npmrc`, тесты, runner pipeline и generated files.
- Можно менять порядок блоков, текст блока, зависимости, scripts.
- Подсказки короче, больше похожи на diagnostics/code review.

---

## 3. Основной экран

- Верхняя панель: курс, урок, режим, запуск, тесты, сохранение.
- Левая зона: библиотека блоков текущей темы.
- Центр: canvas сборки проекта, куда блоки перетаскиваются по шагам.
- Правая зона: сгенерированный код и вкладки внутренних файлов.
- Низ: console/test output, diagnostics, visual state.
- Mobile: те же сущности, но как последовательность `Блоки -> Сборка -> Код -> Запуск -> Ошибка`.

---

## 4. JS-проект первого спринта

Итоговый учебный проект: интерактивный счетчик задач.

Этапы:
1. Вывести заголовок и начальное состояние.
2. Создать массив задач.
3. Написать функцию добавления задачи.
4. Отрендерить список в DOM.
5. Добавить обработчик клика.
6. Добавить простую валидацию.
7. Написать тесты на функции.
8. Перейти в продвинутый режим и изменить generated code руками.

---

## 5. Типы блоков

```json
{
  "id": "array-create",
  "title": "Создать массив задач",
  "category": "data",
  "level": "basic",
  "code": "const tasks = [];",
  "explain": "Массив хранит список задач в памяти.",
  "checks": ["declares:tasks"]
}
```

Блок должен хранить:
- `id`, `title`, `category`, `level`;
- `code` или шаблон кода;
- зависимости от других блоков;
- объяснение для обучающего режима;
- диагностику типовых ошибок;
- тестовые ожидания.

## 6. Runner

### Browser runner
- Выполняет чистый JS через sandbox.
- Перехватывает `console.log`.
- Сравнивает вывод с ожидаемым.
- Показывает DOM-preview для browser lessons.

### Node runner
- Запускается в контейнере или изолированном процессе.
- Поддерживает `npm test`, `npm run build`, `node file.js`.
- Использует lock-файлы и заранее подготовленный cache.
- В вузовском профиле применяет `.npmrc` с сетевыми настройками.

---

## 7. Вузовская сеть

Для учебных компьютеров нужно предусмотреть профиль:

```bash
npm config set strict-ssl false
```

В проектном шаблоне это лучше хранить в `.npmrc` лабораторного окружения:

```ini
strict-ssl=false
registry=https://registry.npmjs.org/
```

Это компромисс для сети вуза. Для домашней и production-среды правильнее настроить корпоративный root CA через `cafile`.

## 8. Минимальная архитектура

- `content`: уроки, блоки, задания, тесты.
- `builder`: drag-and-drop, порядок блоков, параметры блоков.
- `generator`: сборка файлов проекта из блоков.
- `runner`: browser runner, Node runner, test runner.
- `diagnostics`: разбор ошибок и подсказки.
- `accounts`: прогресс, попытки, оплаты, группы.
- `teacher`: кабинеты преподавателей, дедлайны, отчеты.

### 8.1 Архитектура под маленький VPS

Целевой первый сервер: 2 ядра, 3.6 GHz, 10 GB свободного NVMe.

- Один web/API процесс: отдает SPA, API, авторизацию, прогресс, платежные webhooks.
- Одна SQLite база в WAL-режиме: `users`, `courses`, `lessons`, `blocks`, `attempts`, `payments`, `groups`.
- Один runner worker: берет задания из очереди и выполняет их изолированно.
- Browser runner остается на клиенте для простых JS-уроков, чтобы не грузить сервер.
- Node runner включается только для задач с файлами, тестами и npm-зависимостями.
- Concurrency runner по умолчанию: `1`; для 2 CPU можно поднять до `2`, если web/API остается отзывчивым.
- Ограничения job: 5 секунд CPU, 128-256 MB RAM, 1-2 MB stdout, временная папка до 20 MB.
- Сеть внутри runner выключена по умолчанию. Для npm install используем заранее подготовленный cache или отдельный controlled install step.
- Сборка образов идет в GitHub Actions; VPS только скачивает готовый image.

---

## 9. Что нужно доказать прототипом

1. Студент может собрать работающий JS-фрагмент из блоков.
2. Сгенерированный код можно запустить.
3. Ошибка объясняется понятнее, чем стандартный stack trace.
4. Продвинутый режим дает ручное редактирование, а не закрытый no-code.
5. Интерфейс переносится на mobile без потери смысла.

## 10. Модули будущего приложения

```text
apps/
  web/              # React/Vite UI, PWA, desktop/mobile layouts
  api/              # Fastify/Hono API, auth, progress, payments
  runner/           # isolated Node runner and diagnostics
packages/
  content-schema/   # lesson/block/task schemas
  generator/        # blocks -> project files
  diagnostics/      # JS error explanation rules
  ui/               # shared design system
content/
  js/
    lessons/
    blocks/
    projects/
deploy/
  vps/
```

Главное правило: `generator` и `diagnostics` должны быть библиотеками, а не кодом внутри UI. Тогда один и тот же движок будет работать в web, desktop wrapper, mobile wrapper и server-side runner.
