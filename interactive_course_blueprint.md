# Blueprint интерактивного курса из компендиумов

Ревизия: 2026-06-01.

Этот файл фиксирует общую продуктовую структуру, чтобы все компендиумы постепенно превратить в платный интерактивный курс, а не в набор разрозненных конспектов.

---

## 1. Единая структура урока

Каждый урок в платной версии должен иметь одинаковые блоки:
- `Цель`: один измеримый результат, который студент получает за урок.
- `Теория`: короткое объяснение без перегруза справочником.
- `Демо`: минимальный рабочий пример.
- `Практика`: 2-4 задания от простого к прикладному.
- `Автопроверка`: тесты, линтер, snapshot, SQL-assert, CLI-check или проверка контейнера.
- `Разбор ошибок`: типовые ошибки, симптомы и быстрые исправления.
- `Проектный шаг`: маленькое изменение в итоговом проекте курса.
- `Критерии сдачи`: чеклист, по которому можно принять работу.

---

## 2. Типы интерактива

- Кодовые задания: запуск тестов, проверка stdout/stderr, проверка файловой структуры.
- Песочницы: браузерный JS/TS, Python notebooks, SQL playground, Docker sandbox, Linux terminal.
- Диагностика: студент получает сломанный проект и должен найти причину.
- Рефакторинг: улучшить готовый код без изменения поведения.
- Code review: выбрать проблемные места в diff и объяснить риск.
- Мини-проекты: каждую неделю студент собирает часть итогового приложения.

---

## 3. Коммерческая упаковка

- Бесплатный слой: вводные уроки, чеклисты установки, 1-2 практики без проверки.
- Базовый платный слой: все уроки, автопроверка, прогресс, итоговые проекты.
- Pro-слой: ревью кода, разборы ошибок, карьерные задания, сертификат с проверяемым проектом.
- Командный слой: кабинеты преподавателей, дедлайны, отчеты по группам, приватные задания.

---

## 4. Технические требования к платформе

- Контент хранить как Markdown + метаданные урока (`id`, `topic`, `level`, `estimated_minutes`, `stack`, `checks`).
- Каждое задание должно иметь эталонные тесты и скрытые тесты.
- Проверяющая среда должна быть воспроизводимой: Docker image, фиксированные версии SDK и lock-файлы.
- Для прогресса хранить попытки, ошибки, время решения, выполненные критерии.
- Для будущей аналитики тегировать темы: `syntax`, `oop`, `async`, `db`, `testing`, `devops`, `security`, `project`.

---

## 5. MVP

1. Выбрать 3 стартовых направления: Python, JavaScript/TypeScript, Git/Linux.
2. Разбить каждый компендиум на 20-30 уроков с единым шаблоном.
3. Для каждого направления сделать один итоговый проект и 30-50 автопроверяемых заданий.
4. Поднять простую платформу: каталог уроков, editor, runner, progress, auth, payments.
5. После MVP расширить C++, C#, Go, Rust, SQL/NoSQL, Docker, Kotlin, Swift.
