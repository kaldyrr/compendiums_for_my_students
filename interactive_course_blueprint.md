# Blueprint интерактивного курса из компендиумов

Ревизия: 2026-06-01.

Этот файл фиксирует общую продуктовую структуру, чтобы все компендиумы постепенно превратить в платный интерактивный курс, а не в набор разрозненных конспектов.

---

## 1. Единая структура урока

Каждый урок в платной версии должен иметь одинаковые блоки:
- `Цель`: один измеримый результат, который студент получает за урок.
- `Теория`: короткое объяснение без перегруза справочником.
- `Демо`: минимальный рабочий пример.
- `Практика`: 2-4 задания от простого к прикладному.
- `Автопроверка`: тесты, линтер, snapshot, SQL-assert, CLI-check или проверка контейнера.
- `Разбор ошибок`: типовые ошибки, симптомы и быстрые исправления.
- `Проектный шаг`: маленькое изменение в итоговом проекте курса.
- `Критерии сдачи`: чеклист, по которому можно принять работу.

---

## 2. Типы интерактива

- Кодовые задания: запуск тестов, проверка stdout/stderr, проверка файловой структуры.
- Песочницы: браузерный JS/TS, Python notebooks, SQL playground, Docker sandbox, Linux terminal.
- Диагностика: студент получает сломанный проект и должен найти причину.
- Рефакторинг: улучшить готовый код без изменения поведения.
- Code review: выбрать проблемные места в diff и объяснить риск.
- Мини-проекты: каждую неделю студент собирает часть итогового приложения.

---

## 3. Коммерческая упаковка

- Бесплатный слой: вводные уроки, чеклисты установки, 1-2 практики без проверки.
- Базовый платный слой: все уроки, автопроверка, прогресс, итоговые проекты.
- Pro-слой: ревью кода, разборы ошибок, карьерные задания, сертификат с проверяемым проектом.
- Командный слой: кабинеты преподавателей, дедлайны, отчеты по группам, приватные задания.

---

## 4. Технические требования к платформе

- Контент хранить как Markdown + метаданные урока (`id`, `topic`, `level`, `estimated_minutes`, `stack`, `checks`).
- Каждое задание должно иметь эталонные тесты и скрытые тесты.
- Проверяющая среда должна быть воспроизводимой: Docker image, фиксированные версии SDK и lock-файлы.
- Для прогресса хранить попытки, ошибки, время решения, выполненные критерии.
- Для будущей аналитики тегировать темы: `syntax`, `oop`, `async`, `db`, `testing`, `devops`, `security`, `project`.

---

## 5. MVP

1. Начать с JavaScript как самого быстрого направления для визуального прототипа.
2. Сделать desktop-first приложение с окном в стиле macOS и адаптивной мобильной версией.
3. Реализовать два режима: обучающий и продвинутый.
4. Дать студенту проектный маршрут с этапами и готовыми действиями кода, но оставить полный доступ к редактированию исходника.
5. Подключить настоящий runner: код не просто рендерится, а выполняется, тестируется и объясняет вероятную ошибку.
6. После JS MVP расширить TypeScript, Python, Git/Linux, затем C++, C#, Go, Rust, SQL/NoSQL, Docker, Kotlin, Swift.

---

## 6. Формат приложения

- Desktop: основная рабочая поверхность, похожая на аккуратное macOS-окно: маршрут этапов, текущая цель, действия этапа, код, вывод, подсказки.
- Mobile: режим повторения и коротких практик; полноценную сборку проектов оставить доступной, но интерфейс должен складываться в пошаговый flow.
- Обучающий режим: студент идет по этапам проекта, видит только нужные сейчас действия, получает мягкие подсказки и автопояснение ошибок.
- Продвинутый режим: доступны внутренние файлы, `package.json`, `.npmrc`, тесты, конфиги, ручное редактирование кода и изменение generated-проекта.
- Главная идея: это тот же язык программирования, но с режимом конструктора, где сложная инфраструктура автоматизирована, а не спрятана навсегда.

---

## 7. Антистраховый UX

Курс должен быть понятен студенту, для которого фразы `сделай fork`, `запусти Vite`, `создай React app`, `открой App.jsx` звучат как риск что-то сломать.

Правило: сначала действие и результат, потом техническое название.

| Страшная фраза | Что видит студент | Что делает платформа |
| --- | --- | --- |
| Сделай fork | Создать свою копию проекта | Создает workspace/repo copy |
| Запусти Vite | Подготовить React-проект | Генерирует Vite template |
| Установи зависимости | Подготовить инструменты | Выполняет install/cache step |
| Открой `App.jsx` | Изменить главный экран | Открывает нужный файл |
| Сделай commit | Сохранить версию | Создает commit с понятным сообщением |
| Исправь ошибку сборки | Починить запуск | Показывает причину и точку исправления |

Технический термин показываем рядом мелко: `Изменить главный экран (src/App.jsx)`. Так студент привыкает к реальным словам без ощущения, что его бросили в терминал.

Еще одно правило: блоки кода не должны быть главным экраном курса. Главный экран - это маршрут проекта:
- этап с понятной целью;
- короткий набор действий для этой цели;
- запуск и проверка именно этого этапа;
- автоматический переход к следующему этапу после результата.

Дополнительные блоки появляются как продуктовый выбор: "добавить статистику", "добавить фильтр", "показать ошибку нормально". Если блок нельзя объяснить через пользу для проекта, он не должен попадать в обучающий режим.

Анимации нужны только для смысла: подсветить текущее действие, показать завершение этапа, мягко открыть следующий шаг, привлечь внимание к диагностике.

## 8. Вузовская сеть

- Доступ в сеть предполагается, но он может идти через прокси, DPI, локальные сертификаты и ограничения TLS.
- Для npm-заданий в вузовском образе допускается команда:
  ```bash
  npm config set strict-ssl false
  ```
- Это небезопасная настройка для личных машин и production. В идеале вуз должен выдать корневой сертификат и настроить `cafile`; `strict-ssl=false` используем как обязательный учебный workaround только внутри контролируемой лабораторной сети.
- Платформа должна поддерживать шаблон сетевой настройки: `.npmrc`, proxy vars, registry mirror, offline cache, lock-файлы.
- Все runner-образы должны логировать сетевую диагностику: `npm config list`, registry, proxy, Node/npm versions.

## 9. Архитектурный принцип для VPS

Первый production-ready MVP должен быть большим по замыслу, но маленьким по эксплуатации. Целевая минимальная машина: 2 CPU core, 3.6 GHz, 10 GB free NVMe.

- Не начинаем с микросервисов. Используем modular monolith: один web/API процесс, одна БД, один runner worker.
- Frontend: responsive web/PWA с desktop-first рабочей областью и mobile flow. Нативные оболочки можно добавить позже через Tauri/Capacitor.
- Backend: Node.js 24 LTS, Fastify/Hono или аналогичный легкий HTTP слой, строгие boundaries по модулям.
- Storage MVP: SQLite WAL для пользователей, прогресса, попыток и платежных webhooks. PostgreSQL выносится на отдельный сервер/managed service, когда появится нагрузка.
- Static assets отдает Caddy/Nginx; Node не должен тратить CPU на то, что умеет reverse proxy.
- Runner отделен от web/API. Нельзя выполнять код студентов в основном процессе приложения.
- Runner-задачи идут через очередь с concurrency `1` или `2`, лимитом времени, памяти, размера файлов и stdout.
- Сборку Docker-образов делаем в CI, не на VPS. На VPS только pull готового image из registry.
- Логи храним с rotation; без тяжелого ELK/Prometheus стека на первом сервере.
- Private repo: деплой через GitHub Actions + GHCR image или deploy key/read-only token. Секреты не кладем в репозиторий.

---

## 10. Когда масштабироваться

- Больше 20-30 одновременных студентов с runner-задачами: вынести runner на отдельный VPS.
- База больше 1-2 GB или появились тяжелые отчеты: перейти с SQLite на PostgreSQL.
- Нужны файлы проектов, медиа, сертификаты: вынести артефакты в S3-compatible storage.
- Нужны школы/вузы с группами: добавить tenant limits, отчеты преподавателя и отдельные rate limits.
