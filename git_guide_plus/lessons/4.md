# 📋 Описание задания

## Актуализация 2026-06

- В новых командах используйте `git switch -c`, но знайте `git checkout -b`, потому что он часто встречается в существующих инструкциях.
- Для командной разработки показывайте branch protection, PR review, passing checks и signed commits как отдельные критерии зрелости.
- В интерактивной версии урок проверяет ветки, merge/rebase сценарии, разрешение конфликта и чистую историю.

## Цель: Освоить продвинутые техники работы с ветками в Git, включая создание, переключение, слияние, перебазирование и разрешение конфликтов.

## Задачи:

- Научиться создавать и управлять различными типами веток

- Освоить методы слияния (merge) и перебазирования (rebase)

- Научиться разрешать конфликты слияния

- Понять различия между fast-forward и three-way merge

- Практиковаться в работе с удаленными ветками

# 🛠️ Пошаговое выполнение
## Шаг 1: Подготовка репозитория для практики
```bash
# Создаем новый репозиторий для практики
mkdir branch-practice
cd branch-practice
git init

# Создаем начальную структуру
echo "# Branch Practice Project" > README.md
mkdir src
echo "console.log('Hello World');" > src/main.js

# Первый коммит
git add .
git commit -m "chore: initial project structure"
```

## Шаг 2: Создание различных типов веток
```bash
# Создаем feature-ветку для новой функциональности
git switch -c feature/user-authentication

# Добавляем функциональность аутентификации
echo "function login() { return true; }" > src/auth.js
git add src/auth.js
git commit -m "feat: add user authentication"

# Создаем bugfix-ветку
git switch main
git switch -c hotfix/critical-bug

# Исправляем критическую ошибку
echo "console.log('Fixed critical bug');" >> src/main.js
git add src/main.js
git commit -m "fix: resolve critical security issue"

# Создаем ветку для документации
git switch main
git switch -c docs/api-reference

# Добавляем документацию
echo "# API Reference" > docs/api.md
git add docs/api.md
git commit -m "docs: add API reference"
```

## Шаг 3: Просмотр и навигация по веткам
```bash
# Просмотр всех локальных веток
git branch

# Просмотр всех веток (включая удаленные)
git branch -a

# Просмотр веток с дополнительной информацией
git branch -v

# Просмотр веток, которые уже слиты с main
git branch --merged main

# Просмотр веток, которые еще не слиты с main
git branch --no-merged main

# Переключение между ветками
git switch feature/user-authentication
git switch hotfix/critical-bug
```

## Шаг 4: Fast-Forward слияние
```bash
# Переключаемся на main
git switch main

# Убеждаемся, что main позади hotfix ветки
git log --oneline --graph --all

# Выполняем fast-forward merge
git merge hotfix/critical-bug

# Проверяем историю
git log --oneline
```

## Шаг 5: Three-Way Merge (слияние с коммитом слияния)
```bash
# Вносим изменения в main после слияния
echo "// Main branch update" >> src/main.js
git add src/main.js
git commit -m "chore: update main branch"

# Переключаемся на feature ветку и вносим изменения
git switch feature/user-authentication
echo "function logout() { return true; }" >> src/auth.js
git add src/auth.js
git commit -m "feat: add logout functionality"

# Выполняем three-way merge
git switch main
git merge --no-ff feature/user-authentication

# Проверяем историю с коммитом слияния
git log --oneline --graph
```

## Шаг 6: Перебазирование (Rebase)
```bash
# Создаем новую feature ветку
git switch -c feature/payment-system

# Вносим несколько коммитов
echo "function processPayment() { }" > src/payment.js
git add src/payment.js
git commit -m "feat: add payment processing"

echo "function validateCard() { }" >> src/payment.js
git add src/payment.js
git commit -m "feat: add card validation"

# Тем временем в main появляются новые коммиты
git switch main
echo "// New feature in main" >> src/main.js
git add src/main.js
git commit -m "feat: add new main feature"

# Перебазируем feature ветку на актуальный main
git switch feature/payment-system
git rebase main

# Проверяем линейную историю
git log --oneline --graph
```

## Шаг 7: Разрешение конфликтов слияния
```bash
# Создаем конфликтную ситуацию
git switch main
echo "function conflictInMain() { return 'main'; }" > src/conflict.js
git add src/conflict.js
git commit -m "feat: add function in main"

git switch -c feature/conflicting-changes
echo "function conflictInFeature() { return 'feature'; }" > src/conflict.js
git add src/conflict.js
git commit -m "feat: add function in feature"

# Пытаемся слить - получаем конфликт
git switch main
git merge feature/conflicting-changes
```
Результат: Вы увидите сообщение о конфликте:
```bash
Auto-merging src/conflict.js
CONFLICT (add/add): Merge conflict in src/conflict.js
Automatic merge failed; fix conflicts and then commit the result.
```

## Шаг 8: Решение конфликта
```bash
# Проверяем статус
git status

# Открываем файл с конфликтом в редакторе
code src/conflict.js
```
Содержимое файла с конфликтом:
```bash
<<<<<<< HEAD
function conflictInMain() { return 'main'; }
=======
function conflictInFeature() { return 'feature'; }
>>>>>>> feature/conflicting-changes
```
Решаем конфликт вручную:
```bash
# JavaScript
function resolveConflict() { 
    return 'resolved'; 
}
```
```bash
# Добавляем разрешенный файл
git add src/conflict.js

# Завершаем слияние
git commit -m "fix: resolve merge conflict"

# Проверяем историю
git log --oneline --graph
```

## Шаг 9: Работа с удаленными ветками
```bash
# Добавляем удаленный репозиторий (если еще не добавлен)
git remote add origin git@github.com:your-username/branch-practice.git

# Отправляем все ветки на удаленный репозиторий
git push --all origin

# Получаем информацию об удаленных ветках
git fetch origin

# Создаем локальную ветку для отслеживания удаленной
git switch -c feature/remote-tracking --track origin/feature/user-authentication

# Удаляем локальную ветку (после слияния)
git branch -d hotfix/critical-bug

# Удаляем удаленную ветку
git push origin --delete hotfix/critical-bug
```

## Шаг 10: Продвинутые техники работы с ветками
```bash
# Переименование ветки
git branch -m old-branch-name new-branch-name

# Создание ветки из определенного коммита
git switch -c branch-from-commit <commit-hash>

# Просмотр различий между ветками
git diff main..feature/user-authentication

# Просмотр коммитов, которые есть в feature ветке, но нет в main
git log main..feature/user-authentication

# Интерактивное перебазирование (изменение истории)
git rebase -i HEAD~3
```

## Шаг 11: Практика с Cherry-pick
```bash
# Копируем конкретный коммит из одной ветки в другую
git switch main

# Находим хеш коммита, который хотим скопировать
git log --oneline feature/user-authentication

# Копируем коммит в текущую ветку
git cherry-pick <commit-hash>

# Проверяем результат
git log --oneline -3
```

## 🐛 Возможные проблемы и решения
Проблема: Конфликт при rebase
```bash
# Решение: Разрешить конфликт, затем продолжить
git add .
git rebase --continue
# Или отменить rebase
git rebase --abort
```
Проблема: Невозможно удалить ветку
```bash
# Решение: Принудительное удаление (если уверены)
git branch -D branch-name
```
Проблема: Отслеживание неправильной удаленной ветки
```bash
# Решение: Изменить upstream ветку
git branch -u origin/correct-branch
```
Проблема: Merge конфликт слишком сложный
```bash
# Решение: Использовать инструменты для разрешения конфликтов
git mergetool
```
