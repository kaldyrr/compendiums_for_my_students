# 📋 Описание задания

## Актуализация 2026-06

- Для GitHub используйте SSH или fine-grained PAT; классические пароли для HTTPS не подходят.
- В интерактивной версии урок проверяет remote, upstream, отдельную feature-ветку и корректную синхронизацию с исходной веткой.

## Цель: Научиться работать с существующими Git-репозиториями, клонировать проекты, изучать их структуру и вносить первые изменения.

## Задачи:

- Клонировать существующий репозиторий с GitHub

- Изучить структуру проекта и историю коммитов

- Внести изменения в склонированный проект

- Создать новую ветку для разработки

- Синхронизироваться с исходным репозиторием

# 🛠️ Пошаговое выполнение

## Шаг 1: Выбор репозитория для клонирования

Для практики мы будем клонировать тестовый репозиторий. Вы можете выбрать:

1. Публичный репозиторий с GitHub (рекомендуется):
```bash
# Репозиторий для практики
https://github.com/octocat/Spoon-Knife

# Или любой другой открытый репозиторий
https://github.com/microsoft/vscode
```
2. Собственный репозиторий с предыдущего урока

## Шаг 2: Клонирование репозитория
```bash
# Переходим в папку, где будем хранить проекты
cd ~/Documents/Projects  # или любая другая удобная папка

# Клонируем репозиторий через SSH
git clone git@github.com:octocat/Spoon-Knife.git

# Или через HTTPS с Git Credential Manager/fine-grained PAT
git clone https://github.com/octocat/Spoon-Knife.git

# Переходим в папку проекта
cd Spoon-Knife
```
Результат: Вы должны увидеть:
```bash

remote: Counting objects: 100% (X/X), done.
remote: Compressing objects: 100% (X/X), done.
remote: Total X (delta 0), reused X (delta 0), pack-reused 0
Receiving objects: 100% (X/X), done.
```

## Шаг 3: Изучение структуры проекта
```bash
# Просматриваем файлы в проекте
ls -la

# Изучаем структуру репозитория
tree .  # Если команда tree установлена

# Или используем альтернативу
find . -type f -name "*.md" -o -name "*.js" -o -name "*.json" | head -10
```

## Шаг 4: Изучение истории коммитов
```bash
# Просмотр полной истории коммитов
git log

# Компактный просмотр истории
git log --oneline

# Просмотр истории с графиком ветвления
git log --oneline --graph --all

# Просмотр последних 5 коммитов
git log -5

# Просмотр изменений в конкретном коммите
git show <commit-hash>
```
## Шаг 5: Изучение удаленных репозиториев
```bash
# Просмотр подключенных удаленных репозиториев
git remote -v

# Получение подробной информации об origin
git remote show origin

# Проверка всех веток (локальных и удаленных)
git branch -a
```
## Шаг 6: Создание feature-ветки для изменений
```bash
# Создаем и переключаемся на новую ветку
git switch -c my-feature

# Проверяем текущую ветку
git branch
```

## Шаг 7: Внесение изменений в проект
```bash
# Создаем новый файл
echo "# My Contribution" > MY_FEATURE.md

# Добавляем содержание в файл
cat << EOF > MY_FEATURE.md
# My Feature

This is my contribution to the project.

## Changes Made
- Added documentation
- Improved formatting

Created by: $(whoami)
Date: $(date)
EOF

# Или редактируем существующий файл (если есть index.html)
# code index.html  # Открываем в редакторе
```

## Шаг 8: Фиксация изменений
```bash
# Проверяем статус изменений
git status

# Добавляем новые файлы в staging area
git add MY_FEATURE.md

# Проверяем, что будет включено в коммит
git status

# Создаем коммит с описанием
git commit -m "docs: add feature documentation"

# Просматриваем историю с нашим коммитом
git log --oneline -3
```

## Шаг 9: Синхронизация с исходным репозиторием (upstream)
Если вы работаете с форком:
```bash
# Добавляем оригинальный репозиторий как upstream
git remote add upstream git@github.com:octocat/Spoon-Knife.git

# Проверяем добавленный remote
git remote -v

# Получаем последние изменения из upstream
git fetch upstream

# Перебазируем нашу ветку на актуальную main из upstream
git rebase upstream/main
```

## Шаг 10: Отправка изменений в свой форк
```bash
# Отправляем ветку в наш форк на GitHub
git push -u origin my-feature

# Проверяем, что ветка создалась на GitHub
git branch -r
```

## Шаг 11: Практика с разными сценариями клонирования
Клонирование в конкретную папку:
```bash
git clone git@github.com:octocat/Spoon-Knife.git my-custom-folder
```

Клонирование только определенной ветки:
```bash
git clone git@github.com:octocat/Spoon-Knife.git my-custom-folder
```

Клонирование с ограничением глубины истории:
```bash
git clone --depth 1 git@github.com:octocat/Spoon-Knife.git
```

## 🐛 Возможные проблемы и решения
Проблема: Ошибка аутентификации при клонировании
```bash
# Решение: Проверить SSH-ключ
ssh -T git@github.com
# Если не работает, использовать HTTPS
git clone https://github.com/username/repo.git
```
Проблема: Репозиторий слишком большой, клонирование занимает много времени
```bash
# Решение: Клонировать с ограничением глубины
git clone --depth 1 git@github.com:username/repo.git
```
Проблема: Конфликт при rebase с upstream
```bash
# Решение: Разрешить конфликты вручную, затем продолжить
git rebase --continue
# Или отменить rebase
git rebase --abort
```
Проблема: Нет прав для записи в исходный репозиторий
```bash
# Решение: Сделать fork репозитория и работать со своим форком
# На GitHub нажать "Fork", затем клонировать свой форк
git clone git@github.com:YOUR-USERNAME/repo-name.git
```
## 💡 Полезные команды для исследования
```bash
# Просмотр изменений в рабочей директории
git diff

# Просмотр изменений в staged файлах
git diff --staged

# Просмотр кто последний менял файл
git blame README.md

# Поиск в истории коммитов
git log --grep="fix"  # поиск коммитов с "fix" в сообщении

# Просмотр изменений в конкретном файле за время
git log -p -- README.md
```
